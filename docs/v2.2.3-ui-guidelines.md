# V2.2.3 企业工作台界面规范

本文记录 V2.2.3 的设计语言、页面状态、响应式、无障碍和 UI 测试约束。它服务于 Enterprise Agent Hub 的长期维护，不建立第二套前端技术栈。

## 1. 产品定位

界面首先是企业内部持续工作的 Agent 工作台，其次才是演示产品。设计目标是专业、克制、可信、清晰和高信息密度。重要操作应在短时间内可定位，状态和数据来源必须明确，视觉优化不得改变业务语义。

仓库内 `redesign-existing-projects` Skill 已用于 Scan、Diagnose、Fix 审查。采纳内容包括视觉层级、对齐、组件一致性、交互反馈、空/错/降级状态、响应式和可访问性；拒绝外部字体、营销式超大标题、巨大留白、破格不对称、玻璃拟态、纹理、夸张渐变、滚动揭示、弹簧动画和全项目重写。

## 2. 设计 Token

Token 位于 `src/app/globals.css` 与 `tailwind.config.ts`：

- 背景：应用背景、内容表面、次级表面；
- 文字：主文字、次级文字、弱化文字；
- 结构：边框、分割线、面板圆角、控件圆角、页面间距；
- 状态：品牌信息、成功、警告、错误和中性状态；
- 交互：统一焦点环、按下反馈、快速/常规动效时长；
- 数据：指标和表格数字使用 tabular figures；
- 内容宽度：全局工作区最大宽度为 1440px，Chat 保持独立全高区域。

外部字体、图标库、UI 框架和图表依赖未增加。系统字体优先覆盖中文与企业环境。

## 3. 通用组件

- `PageHeader`：页面标题、说明、操作和安全元数据；
- `StatusBadge`：状态文字与非颜色标记并存；
- `StatePanel`：Loading、Empty、Error、Warning、Success、Degraded、Conflict 等统一结构；
- `SectionCard`：有明确层级的内容分区，避免无意义卡片嵌套；
- `DataTableShell`：标题、工具栏和局部横向滚动；
- `ResponsiveToolbar`：桌面紧凑排列、小屏纵向可达；
- `ConfirmDialog`：危险操作确认、焦点锁定、Escape 关闭和焦点恢复；
- `CollapsibleSection`：使用 `aria-expanded` / `aria-controls` 显示技术详情。

通用组件只负责展示与交互语义，不直接访问 API、Repository、localStorage 或数据库。

## 4. 页面状态规范

状态必须由图形/文字/语义共同表达，不能只依赖颜色。

| 状态 | 语义 | 推荐操作 |
| --- | --- | --- |
| Loading | `aria-busy` 或礼貌 live region | 不提供会产生重复提交的操作 |
| Empty | 说明当前为空及下一步 | 指向真实创建/导入/运行入口 |
| Error | 安全错误文案，必要时 `role=alert` | 重试或返回稳定页面 |
| Degraded | 警告色与“只读/暂不可用”文字 | 禁止伪装成功的写入，允许重试连接 |
| Conflict | 明确说明 409/CAS 冲突 | 重新读取最新版本后再试 |
| Rate limited | 区别于普通失败 | 稍后重试，不展示内部限流配置 |
| Unauthorized / Forbidden | 区分缺少口令与口令错误 | 重新输入，不回显口令 |
| Import processing | 只展示服务端真实统计 | 允许刷新、取消或等待 |
| Partial failed | 显示成功/失败/跳过/冲突计数 | 只重试允许重试的失败项 |
| Real API unconfigured | 明确未配置 | 保留开发模拟模式，不伪装真实生成 |

后端堆栈、数据库地址、完整正文、Prompt、Summary、Cookie 和 Secret 不得进入状态文案。

## 5. 响应式规范

必须验证 1440、1280、1024、768 和 390px：

- 页面不得产生整体横向溢出；表格和代码块在自身容器内滚动；
- 390px 下主导航可横向访问并将当前页滚动到可见范围；
- Chat 会话列表使用带焦点锁定的移动抽屉，Composer 不遮挡最后一条消息；
- Knowledge 主从信息在桌面保持高密度，小屏将详情和操作按内容顺序展开；
- 批量导入文件、预览项、冲突处理和任务操作在窄屏仍可到达；
- 弹窗最大高度受视口约束，内容区可滚动；
- 触摸按钮和表单控件的最小高度保持约 40px。

## 6. 无障碍

- 根布局提供“跳到主要内容”链接，主区域可被程序聚焦；
- 导航使用 `aria-current=page`；
- 标题层级和 section label 必须连续、明确；
- 表单使用可见 label 或稳定的 `aria-label`，错误信息通过 `aria-describedby` / live region 关联；
- 图标或短文本按钮必须有可读名称；
- Dialog 使用 `role=dialog/alertdialog`、`aria-modal`、焦点锁定、Escape 和关闭后焦点恢复；
- 进度、复制、保存、错误和流式完成使用克制的 `aria-live`；逐 delta 正文不应重复朗读；
- 状态不只靠颜色，表格使用 caption 和 `scope`；
- 全局提供可见 `focus-visible`；
- `prefers-reduced-motion: reduce` 下取消平滑滚动并将非必要动画/过渡缩短到近零。

目标是常规 WCAG AA 可用性，不宣称已获得正式认证。

## 7. UI 测试与视觉检查

```powershell
npm.cmd run test:ui
npm.cmd run typecheck
npm.cmd run build
npm.cmd run e2e
```

Vitest 覆盖通用组件的安全状态、语义关联、Token 与 reduced-motion；Playwright 覆盖当前导航、Skip Link、Chat、Knowledge、导入、Evaluation、Ops、Tools 以及 390px 无横向溢出。现有 `data-testid`、按钮名称和业务表单语义必须保留。

视觉检查至少包含首页、Chat 空/生成/完成/错误、Knowledge、知识包、导入预览/处理中/部分失败、Evaluation、Ops、Tools、About，以及 390px Chat/Knowledge。截图、录像、`test-results` 和 `trace.zip` 只用于本地审查，不提交 Git。

## 8. 稳定边界

V2.2.3 不修改 Router、RAG、Tool Calling、Real API、NDJSON、Completion Fence、Context Manager、Rolling Summary、Summary Patch、CAS、Repository、Workspace Cookie、Prisma Schema、migration、claim/lease、解析和分块逻辑。Node 保持 20.19.5，Prisma 保持 6.19.3，Undici 保持 7.28.0。

腾讯云未部署 V2.2.3，线上仍为 V2.0.4。
