# V2.2.2 生产加固与启动检查

## 目标与边界

V2.2.2 是依赖兼容、导入任务、PostgreSQL 查询和故障恢复的生产加固版本，不增加新的业务场景或主界面。Node 固定为 20.19.5，Prisma / Prisma Client 固定为 6.19.3；不包含 Node 22、COS、OCR、向量数据库、账号权限或长期记忆。

本版本 Git 发布不等于腾讯云部署。腾讯云继续运行 V2.0.4，生产 migration、Release、Canary 和 PM2 切换必须获得独立授权。

## Node 20 与 undici

V2.2.1 直接依赖的 `undici@8.5.0` 要求 Node 22.19.0 以上，且在 Node 20.19.5 的代理 Real API 路径会因缺少 WebIDL 能力实际失败。V2.2.2 将直接依赖精确固定为 `undici@7.28.0`：

- 官方 engine 为 Node `>=20.18.1`，覆盖项目基线 20.19.5。
- 保留当前使用的 `fetch`、`ProxyAgent` 和 dispatcher API。
- 不使用 `overrides`、`--force` 或 `--legacy-peer-deps`。
- 无代理请求继续使用 Node 20 内置 `fetch`；配置代理时才加载项目依赖。
- Real API 每个顶层调用复用一个 ProxyAgent，并在完成、失败、超时或取消后关闭。

解除精确固定的条件：项目未来单独升级 Node 基线，并在新基线上重新执行非流式、流式、代理、AbortSignal、超时和错误响应门禁。GitHub Actions 自身内部 Node 运行时不属于项目运行依赖，不能用它作为升级本项目 Node 的理由。

## 集中资源限制

生产限制统一定义在 `src/lib/knowledge/import-limits.ts`，客户端提示不能覆盖服务端最终限制：

| 限制 | 值 |
| --- | ---: |
| 单批文件 | 10 |
| 单文件体积 | 5 MiB |
| 单批体积 | 25 MiB |
| 提取字符 | 120,000 |
| PDF 页数 | 200 |
| 分块 / 预览分块 | 500 / 40 |
| DOCX entries / 解压总量 | 500 / 50 MiB |
| DOCX 单 entry / 压缩比 | 10 MiB / 100 |
| 单批解析并发 | 2 |
| 单进程预览请求 / 同工作区 | 2 / 1 |
| 单文件解析 / 预览整体 | 10 秒 / 60 秒 |
| 单项处理 / 处理会话 | 20 秒 / 90 秒 |
| 最大重试 | 3 |
| claim lease / 续租间隔 | 30 秒 / 10 秒 |
| 临时正文和未确认预览保留 | 24 小时 |

保留期清理由 PostgreSQL Repository 在预览、任务读取/恢复、处理和重试入口执行。过期且尚需正文的 Item 使用安全错误码 `knowledge_import_temporary_content_expired` 进入失败状态，并清除正文、claim 与 lease；任务计数和 revision 在同一事务更新。终态仅移除残留正文，不删除正式知识文档或分块。

`formData()` 仍受 25 MiB 请求硬上限保护；解析按受限并发逐文件进行。PDF 逐页检查取消、页数和提取字符边界；DOCX 在交给 mammoth 前检查 ZIP 路径、条目数、单项/总解压大小、压缩比、加密和危险结构。mammoth 不提供可硬终止的解析 API，因此本版以严格 ZIP 预检、较低并发和外层 deadline 限制风险；进程级解析隔离留给后续专版。

## claim、lease 与恢复

- 领取使用随机不可预测 token、Item revision 和数据库 Serializable 事务。
- READY 或 lease 已过期的 PROCESSING Item 才可领取；未过期 claim 不能被抢占。
- 正式写入前续租，并校验工作区、任务、Item、claim token、状态和 revision。
- 旧 token 在重领后不能提交；成功 Item 不能再次领取。
- retry 只重置仍有临时正文且未达到 3 次上限的失败项。
- cancel 清除未完成项的 claim 和临时正文，已成功文档不回滚。
- 处理器中断后由 lease 到期恢复；新处理器实例只能产生一个最终文档和一套 chunks。
- 单个文档、chunks、Item 状态和 Job 统计在一个事务中完成；网络丢失后的重复 process 不会重复导入。

## 生产启动检查

使用项目 Node 20.19.5 执行：

```bash
npm run production:check
```

脚本只输出布尔值、版本、存储模式和安全状态，不输出环境变量值、数据库连接串、Cookie、Prompt 或正文。它检查：

- Node 是否精确为 20.19.5；
- Undici、Prisma Client、PDF、DOCX 与 ZIP 解析器能否加载；
- 临时目录是否可安全写入和清理；
- server storage 开启时，数据库与高熵 Session Secret 是否就绪；
- V2.2.2 migration 是否成功完成，关键字段是否存在；
- Real API 与 Ops 口令是否已配置（只报告 configured，不主动输出或探测 Secret）。

任一必需检查失败时返回非零退出码。local 模式允许数据库未配置；server storage 已开启但数据库、Secret、migration 或 Schema 不完整时必须报告 degraded 并失败。

## 安全健康接口

`GET /api/health` 返回：

- `applicationHealthy`
- `databaseConfigured`
- `databaseHealthy`
- `migrationReady`
- `storageMode`
- `realApiConfigured`
- `realApiHealthy`（未显式探测时为 `null`）
- `parserReady`
- `nodeCompatible`
- `version`

该接口不会为了健康检查消耗模型额度。需要显式探测 Real API 时继续使用既有 `POST /api/llm/health`；两者均不得返回 Key、代理地址或完整响应正文。

## 压力与故障门禁

- `test:real-api:node20`：本地假 HTTP 服务验证非流式、流式、代理、AbortSignal、超时和上游错误。
- `test:import:hardening`：资源边界、claim 竞争、续租、重领、retry/cancel 和安全错误。
- `test:import:stress`：使用仓库内小型内存 fixture 执行两组 10 文件压力：一组正文接近实际分块上限，另一组混合 TXT、Markdown、文本型 PDF 与 DOCX；同时验证解析超时/取消的安全错误码。
- 压力组合会并发提交同一工作区的重复 process 请求，确认最终只写入 1 份文档；两个独立工作区可同时各完成 1 份文档。精确重复的 `replace` 只执行一次文档替换和旧 chunk 清理，终态重放不会重复写入；取消任务后文档和 chunk 写入数保持为 0。
- 压力测试只在测试进程内统计项目数、成功/失败数、耗时和 RSS 峰值增量，并采用宽松安全上限避免 CI 抖动；不生成大型 fixture，不连接真实数据库，也不打印正文、文件路径或运行凭据。
- `test:migration:v222`：独立 PostgreSQL 16 上验证 V2.2.1 升级、数据保留和重复 deploy。
- `production:check`：启动前 fail-closed 检查。

压力输出仅包含数量、状态、安全错误码、耗时和内存统计，不包含正文、文件完整路径、Cookie、Secret 或数据库 URL。
