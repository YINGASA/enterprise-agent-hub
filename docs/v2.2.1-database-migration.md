# V2.2.1 数据库迁移说明

## 迁移范围

正式 migration：`prisma/migrations/20260717000000_v221_knowledge_pack_import/migration.sql`。

迁移在 V2.2.0 PostgreSQL Schema 上追加：

- `KnowledgePackStatus`、`ImportItemStatus` 枚举。
- `knowledge_packs`、`import_items` 表。
- `knowledge_documents` 的内容 checksum、revision、企业知识包、文件来源和导入任务字段。
- `import_jobs` 的知识包、幂等键、进度计数、revision、完成时间和耗时字段。
- 工作区复合外键、唯一约束和检索索引。

迁移不会删除 Workspace、Conversation、Message、KnowledgeDocument、KnowledgeChunk、ImportJob、StorageMigration 或 V2.2.0 业务数据。旧知识文档允许新增字段为空；旧单文档任务会保留，并补齐安全的计数默认值。

## 验证顺序

```bash
npm run db:generate
npm run db:validate
npm run db:migrate:deploy
npm run db:status
npm run test:migration:v221
```

发布门禁需要分别验证：

1. `test:migration:v221` 创建独立临时数据库，仅部署 V2.2.0 migration，写入旧知识文档、chunk 和任务，再部署 V2.2.1 并断言数据及默认字段，最后删除临时数据库。
2. 在全新的 PostgreSQL 16 空库执行全部 migration。
3. 运行工作区隔离、知识包外键、任务 claim/lease、replace 事务和 RAG 回归测试。

只允许使用隔离测试数据库；不要使用生产 `DATABASE_URL`，不要执行 `prisma migrate reset`，不要输出连接字符串。

## 回滚策略

应用回滚优先采用前向修复：停止创建或推进新批量任务，保留新增表和字段，再部署兼容旧读取路径的上一应用版本。由于 PostgreSQL 枚举值和成功导入的文档属于已提交业务数据，不应通过破坏性 SQL 自动删除。

如必须做数据库级回退，需要先备份并单独审计：保留已导入文档与 chunks、确认没有活跃导入任务、导出任务结果，再由数据库管理员执行人工回滚方案。本仓库不提供自动 destructive down migration。

## 云端状态

V2.2.1 Git 发布不执行腾讯云数据库 migration，也不创建服务器 Release、Canary 或 PM2 切换。腾讯云仍运行 V2.0.4；生产迁移必须在用户另行授权后，按独立备份、测试库演练和回滚计划执行。
