# V2.2.2 PostgreSQL 备份、迁移与恢复

本文只描述未来获得生产部署授权后的操作顺序。本次 V2.2.2 Git 发布不得连接腾讯云数据库，也不得执行生产备份、migration、Canary 或 PM2 操作。

## Release 与 Schema 兼容关系

| 应用版本 | 最低数据库 migration |
| --- | --- |
| V2.2.0 | `20260716000000_v220_server_storage` |
| V2.2.1 | `20260717000000_v221_knowledge_pack_import` |
| V2.2.2 | `20260718000000_v222_production_hardening` |

V2.2.2 migration 只增加规范化查询字段和索引，不删除会话、消息、知识文档、chunks 或导入任务数据。旧应用可能忽略新增字段，但应用回滚前仍必须评估数据库向前兼容性。

目标 PostgreSQL 16 数据库必须使用 UTF-8 编码；规范化回填使用 PostgreSQL `NORMALIZE(..., NFKC)`，非 UTF-8 数据库不满足部署前置条件。

## 部署前备份

1. 记录当前应用 Release、Git commit、Stable Tag、PM2 进程和 Schema migration 状态。
2. 使用数据库供应商支持的一致性备份方式创建备份；命令中的连接信息从安全运行环境注入，不写入脚本、Git、终端截图或报告。
3. 在隔离恢复库完成一次 restore 演练，核对 Workspace、Conversation、Message、KnowledgeDocument、KnowledgeChunk、ImportJob 和 ImportItem 数量。
4. 验证备份加密、保留周期和访问权限。
5. 保留上一应用 Release，不删除历史 Stable Tag。

## Migration 顺序

```bash
npm ci
npm run db:generate
npm run db:validate
npm run db:status
npm run production:check
npm run db:migrate:deploy
npm run db:status
npm run production:check
```

生产环境禁止 `prisma migrate reset`。在 deploy 前必须先在独立 PostgreSQL 16 测试库执行全新部署和 V2.2.1 → V2.2.2 升级。应用启动脚本只有在 migration 与 Schema 检查通过后才可继续。

## Migration 失败

- 立即停止新 Release 启动和流量切换。
- 不重复执行未知的手工 DDL，不修改 `_prisma_migrations`。
- 保留失败日志的安全错误码和 migration 名称；不得复制连接串或数据库原始堆栈到公开报告。
- 由数据库负责人确认事务边界和数据状态。若需恢复，使用已验证的部署前备份恢复到独立实例，核对后再决定生产恢复步骤。
- 未达到 V2.2.2 Schema 时，`production:check` 和 `/api/health` 必须失败或显示 migration 未就绪。

## Canary 与 PM2 检查清单

获得单独授权后才可执行：

1. 在独立 Release 目录安装锁定依赖并生成 Prisma Client。
2. 执行 `production:check`，确认 Node 20.19.5、parser、数据库、migration 和 storage mode。
3. 保留旧 PM2 进程，启动隔离端口 Canary。
4. 验证 local/server/degraded、Real API 非流式/流式、批量导入、lease 恢复、RAG、Ops 401 和两个 Workspace 隔离。
5. 人工终止一个测试处理器，确认 lease 到期后任务只恢复一次。
6. 模拟数据库短暂不可用，确认写操作明确失败；恢复后刷新 revision 再重试，不以本地缓存覆盖服务端。
7. 仅全部通过后评估 PM2 流量切换。

## 应用回滚

- 切回上一 Release 前先停止新写入，记录 V2.2.2 期间新增的任务和数据。
- 不回滚或删除数据库 migration，不执行 reset。
- V2.2.1 可忽略 V2.2.2 的新增规范化字段和索引；回滚后不得依赖 V2.2.2 的续租/重试上限行为处理仍在运行的任务。
- 对 PROCESSING Item，等待 lease 到期或按经审查的恢复流程处理，禁止直接伪造 completed。
- localStorage 数据与旧 key 不删除。

## 数据库故障恢复

- 应用启动数据库不可用：storage mode 为 degraded，写操作失败，不静默落回 localStorage。
- 运行中连接中断、查询超时、事务失败或连接池耗尽：前端只获得安全分类，事务不得部分提交。
- 数据库恢复：重新获取服务端最新 revision、任务状态和 lease；用户显式重试未完成操作，不自动以浏览器缓存覆盖服务端。
- 安全错误类别仅限 `unavailable`、`timeout`、`conflict`、`transaction_failed`、`pool_exhausted`、`unknown_storage_error`。

## 导入任务恢复与临时数据

- 未确认预览和临时 extracted text 保留最多 24 小时。
- 已取消任务不会恢复；完成项不会重新领取。
- 失败项最多重试 3 次，超过上限保持最终失败。
- 不保存原始上传文件；若未来使用临时文件，必须放入系统临时目录，以随机名称创建，并在完成、失败、超时和取消后清理。
- Ops 和日志只记录聚合计数、安全枚举和耗时，不记录正文或完整文件路径。
